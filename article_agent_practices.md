# 从 Ward Agent 重构谈 Agent 设计的几个关键思路

最近把 [Ward Agent](https://github.com/rainj2013/ward-agent) 从手写的 agent loop 重构到了 Mini-Agent 框架。重构过程中花了很多时间在代码组织上，但真正让我收获更大的，是一些**产品层面**的思考——比如上下文复用、工具的职责边界、以及用户感知层面的细节。这篇文章不聊太多代码实现，主要聊聊这些设计上的思路。

---

## 1. 上下文复用：同一份数据，用两次

Ward 有一个很常见的使用场景：

用户在行情页面点了"标普500"，右侧智能问答问"最近3天走势怎么样"。

这个问题拆开看，需要两个信息：**标普500的近期K线数据**、**对这些数据的解读**。但这里有个细节——页面本身已经在渲染标普500的数据了（顶部卡片会显示行情）。

那么问题来了：**AI 回答时，是重新调一次工具去拿数据，还是直接用页面上已有的数据？**

两种做法各有问题：

- **直接用页面数据**：AI 拿到的是"渲染后的值"，不是原始的 OHLCV 数据，分析能力受限。而且页面数据格式和工具返回格式不一致，容易出 bug。
- **每次都重新调工具**：用户体验没问题，但浪费 API 调用，响应也更慢。

Ward 现在的策略是**优先复用，缺失时自动补齐**：

对话开始时，系统会主动把当前页面的上下文（当前查看的股票/指数、相关数据）注入到 system prompt 里。AI 在回答时，如果发现自己需要某个数据，会先看看上下文里有没有——有就直接用，没有才调工具。

这个逻辑不需要用户在提问时额外说明，AI 自动感知、自己判断。**"自动"二字是用户体验的关键。**

## 2. 工具描述即约束：说清楚边界，比说清楚功能更重要

早期的 Ward 有一个 bug：问"标普500最近怎么样"，AI 会调 `get_stock_kline(symbol="SPX")`——然后报错，因为 SPX 不是股票代码，yfinance 查不到。

问题出在工具描述上。原来只写了"获取个股K线数据"，没有写"不适用于指数"。AI 看到"标普500"和"K线"，觉得匹配，就调了。

修改后的描述变成了：

> "获取个股K线数据。此工具仅适用于个股（如 AAPL、TSLA），**不适用于指数**。"

同时在 system prompt 里加了一条规则：

> "个股用 stock 工具，指数用 index 工具。get_stock_kline 不能用于指数，get_index_kline 不能用于个股。"

这不是代码层面的约束，是**描述层面的约束**。描述写清楚边界，AI 自己就不会越界。

这个教训延伸出一个更大的原则：**工具的职责边界要小而清晰**。与其做一个大而全的工具，不如做多个小工具，每个工具有明确的输入输出和适用场景。AI 调用工具时，判断成本低，出错概率也低。

---

## 3. 渐进式输出：让用户感觉"快"，比真的"快"更重要

SSE 流式输出时，有一个体验细节：工具调用结果返回后，页面会显示"正在查询 get_index_analyze…"。

问题是，这个 spinner 会一直挂着，直到 AI 报告生成完毕。生成报告要调 LLM，平均 30 多秒。

30 秒对用户来说是很漫长的等待。更糟糕的是，spinner 一直挂着，会让用户怀疑系统是不是卡死了。

解决方案是**分步反馈**：

工具开始执行时，前端立即显示一个 loading 状态（带工具名）。工具执行完成、结果返回时，前端收到事件，loading 消失、结果显示出来。这时候 AI 可能还在"思考"——但用户看到结果已经在"等待"了，主观上就觉得快。

本质上是**把"执行中"的感知提前**，而不是等所有东西都就绪才一股脑展示。

同样的思路也适用于 thinking 内容。很多产品选择不展示 thinking，因为用户不关心过程，只关心结果。但 Ward 保留了 thinking，原因是市场分析有信息量——AI 展示它的分析逻辑，用户能学到东西，而且会觉得"这确实是AI在思考，不是模板回复"。

展示思考过程是一种信任建立。

---

## 4. 取消能力：用户要能随时喊停

Agent 执行一个大任务时，用户可能中途改变主意。Ward 支持按 Esc 取消。

实现上，取消动作有几个设计点：

**第一，取消要在安全点检查，而不是任意位置。** 如果在 LLM 调用过程中强行中断，消息历史会留下半截内容，下次对话就会看到一堆不完整的中间状态。Ward 的做法是在每一步开始、每个工具执行前、每个工具执行后设检查点——这些都是自然断点，不破坏状态一致性。

**第二，取消后要清理脏状态。** Ward 的做法是找到最后一个完整的 assistant 消息，把之后的内容全部回滚。这样取消后的消息历史是干净的，用户继续对话不受影响。

**第三，取消要给出明确反馈。** 用户按了 Esc，系统要告诉用户"已取消"，而不是无声无息地停在那里。

## 5. 长对话的自我管理：不要让用户操心

如果一个对话持续很久，LLM 的 context window 是有限的。粗暴的解决方案是设一个固定上限，超过就拒绝对话。

Ward 的做法是**自动摘要**：

当历史消息的 token 数量超过阈值（比如 8 万），系统会自动把之前几轮的内容压缩成一段摘要，替换掉原始的详细内容。压缩策略是：保留用户的原始提问，把 AI 执行过程（调了哪些工具、得到什么结论）压缩成一段话。

对用户来说，这个过程是无感的。TA 不会收到任何"即将压缩历史"的通知，也不需要做任何操作。下次提问时，AI 看到的上下文已经是被压缩过的，但用户自己的话是一字没动过的。

这里有个设计细节：**用户说的话不压缩，AI 的执行过程压缩。** 用户意图是上下文的核心，不能损失。AI 做了什么（中间调了哪些工具、返回了什么中间结果）相对次要，压缩掉没问题。

---

## 6. 快速查询 vs 深度分析：两个入口，两种体验

Ward 里的工具分为两类：

- **快速工具**：如 `get_stock_quote`（实时行情）、`get_index_kline`（K线数据）。这些调完直接返回原始数据，毫秒级响应。
- **深度工具**：如 `get_stock_analyze`、`get_index_analyze`。这些会调 LLM 生成一段分析报告，需要几十秒。

用户在不同场景下需要不同深度。如果只是想知道"当前价格"，等 30 秒生成报告是过度服务。如果想了解"最近走势分析和投资建议"，只给一个收盘价又不够。

所以设计产品时，**不要让用户为每一次查询付出等价的等待成本**。快速问题快速回答，复杂问题再花时间。用户有权选择他们需要的体验深度。

另一个细节：两种工具返回的内容格式不同。快速工具返回结构化数据，前端可以渲染成表格、走势图。深度工具返回自然语言，前端渲染成报告卡片。**产品形态要匹配数据形态。**

---

## 7. 容错与失败：说清楚失败，比假装成功更重要

工具调用会失败——网络超时、数据源挂了、参数不对。

很多产品面对失败会选择"沉默"，或者显示一个模糊的错误信息（比如"网络错误，请稍后重试"）。Ward 的做法是把失败信息**原样返回给 AI**，让 AI 根据失败信息自己决定怎么应对。

比如用户问"帮我分析英伟达"，但 `get_stock_analyze` 因为数据源超时失败了，AI 收到的是：

```
Tool execution failed: ConnectionError: Failed to connect to data source
```

AI 的反应不是给用户一个通用的错误提示，而是会根据这个具体错误判断——可能是数据源临时故障，可以换个数据源试试，或者告诉用户"数据源暂时不可用，建议稍后再试"。

**把错误信息当作上下文的一部分，而不是当作需要被隐藏的东西。** 这样做的好处是 AI 有能力做精准的错误处理，而不是每次失败都给用户返回一个千篇一律的"操作失败"。

---

## 总结

回头看这些设计思路，最核心的一条原则是：

> **把复杂度留在系统内部，把简单留给用户。**

上下文复用、渐进输出、自动摘要、取消清理——这些用户都感知不到，但他们能感觉到"这个东西用起来很顺"。反过来说，如果这些没做好，用户感知到的是"这东西总是卡"、"等了半天没反应"、"换个问法就不行了"。

Agent 产品做得好不好，不在于 AI 有多聪明，而在于细节打磨得有多细致。每一个"感觉不对劲"的时刻，都是设计可以改进的地方。
